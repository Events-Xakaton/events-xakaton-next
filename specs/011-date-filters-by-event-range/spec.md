# Feature Specification: Date Filters by Event Time Range

**Feature Branch**: `011-date-filters-by-event-range`  
**Created**: 2026-03-15  
**Status**: Implemented (build-validated on frontend)  
**Input**: User description: "Фильтр `Сегодня` сейчас учитывает только старт. Нужно учитывать многодневные ивенты и применить это ко всем фильтрам. Проверить и скорректировать `Предстоящие / Прошедшие`."

## Clarifications

### Session 2026-03-15

- Q: Что считать попаданием в day-фильтр? → A: Попадание по пересечению интервала ивента и интервала фильтра.
- Q: Если `endsAtUtc` отсутствует в payload? → A: Фронт использует fallback `eventEnd = eventStart` (legacy-safe).
- Q: Куда относить текущие (ongoing) многодневные ивенты в профиле? → A: В `Предстоящие`, а не в `Прошедшие`.

## User Scenarios & Testing _(mandatory)_

### User Story 1 - Day filters include overlapping multi-day events (Priority: P1)

Как пользователь ленты, я вижу ивент в фильтре дня, если ивент пересекается с выбранным периодом, даже если начался раньше.

**Independent Test**: проверить `Сегодня/Завтра/Неделя/Выходные/След. нед.` на ивенте с диапазоном `старт вчера -> конец завтра`.

**Acceptance Scenarios**:

1. **Given** ивент идет с вчера до завтра, **When** выбран фильтр `Сегодня`, **Then** ивент отображается.
2. **Given** ивент идет с сегодня до послезавтра, **When** выбран фильтр `Завтра`, **Then** ивент отображается.
3. **Given** ивент начинается до `На этой неделе`, но заканчивается внутри окна, **When** выбран `На этой неделе`, **Then** ивент отображается.
4. **Given** ивент не пересекает окно фильтра, **When** выбран соответствующий фильтр, **Then** ивент не отображается.

---

### User Story 2 - Profile tabs `Предстоящие / Прошедшие` classify by event end (Priority: P1)

Как пользователь профиля, я вижу длительные ивенты в правильной вкладке: пока не завершились — в `Предстоящие`, после завершения — в `Прошедшие`.

**Independent Test**: проверить ивент `старт вчера, конец завтра` и ивент `конец вчера` в профильных вкладках.

**Acceptance Scenarios**:

1. **Given** ивент уже начался, но еще не завершился, **When** открыта вкладка `Предстоящие`, **Then** ивент присутствует.
2. **Given** ивент завершился, **When** открыта вкладка `Прошедшие`, **Then** ивент присутствует.
3. **Given** ивент уже начался, но еще не завершился, **When** открыта вкладка `Прошедшие`, **Then** ивент отсутствует.

---

### Edge Cases

- `endsAtUtc` отсутствует или невалиден: использовать `startsAtUtc` как end-границу.
- Невалидный `startsAtUtc`: ивент не должен падать по runtime, допускается fallback-показ как сейчас.
- Неконсистентные данные (`endsAtUtc < startsAtUtc`): трактовать как мгновенный ивент в `startsAtUtc`.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: Система MUST определять попадание ивента в day-фильтр через пересечение интервала ивента `[startsAtUtc, endsAtUtc)` и интервала фильтра.
- **FR-002**: Система MUST применить логику пересечения ко всем day-фильтрам `Сегодня`, `Завтра`, `На этой неделе`, `В эти выходные`, `На следующей неделе`.
- **FR-003**: Система MUST поддерживать legacy-поведение для payload без `endsAtUtc` через fallback `eventEnd = eventStart`.
- **FR-004**: Система MUST относить активные (не завершенные) ивенты в профиле во вкладку `Предстоящие`.
- **FR-005**: Система MUST относить ивенты в профиле во вкладку `Прошедшие` только после `endsAtUtc <= now`.
- **FR-006**: Backend `/events` list MUST отдавать `endsAtUtc` в item payload для корректной клиентской фильтрации длительных ивентов.

### Key Entities _(include if feature involves data)_

- **EventTimeRange**: `startsAtUtc` + `endsAtUtc?` c fallback/normalization.
- **FilterWindow**: вычисляемое локальное окно для day-фильтра.
- **ProfileEventBucket**: фронтовая классификация `upcoming | past` на основе `endsAtUtc`.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: Для 100% smoke-кейсов day-фильтры включают пересекающиеся многодневные ивенты.
- **SC-002**: Для 100% smoke-кейсов ивенты `ongoing` не попадают в `Прошедшие` в профиле.
- **SC-003**: Для 100% payload без `endsAtUtc` фильтрация выполняется без runtime ошибок.
