# Feature Specification: Event Location Link Rendering and Google Maps Handling

**Feature Branch**: `009-event-location-maps-link`  
**Created**: 2026-03-15  
**Status**: Draft  
**Input**: User description: "Если в ивенте в поле Локация вписать длинную ссылку, то это выглядит криво. Нужно, чтобы ссылка корректно помещалась в поле. Плюс, нужно распознавание гугл ссылки. Если это ссылка на гугл карты, по тапу надо открывать либо приложение гугл мэпс либо по ссылке."

## Clarifications

### Session 2026-03-15

- Q: На каких экранах обязательна новая логика отображения/тапа по локации? → A: На экране `EventDetails` в блоке `Детали -> Локация` (MVP), без изменения бэкенд-контрактов.
- Q: Что считать "Google Maps ссылкой"? → A: URL с хостом `maps.app.goo.gl`, `maps.google.*`, либо `google.*` с путем `/maps`.
- Q: Какой UX для очень длинных URL в строке локации? → A: Локация должна оставаться внутри карточки без выезда за границы; текст показывается максимум в 2 строки с корректным переносом и троеточием.
- Q: Нужен ли отдельный fallback, если Google Maps app не открывается? → A: Да, обязательно открывать HTTPS-ссылку (внешний браузер/вебview) как fallback.

## User Scenarios & Testing _(mandatory)_

### User Story 1 - Корректный рендер длинной ссылки в локации (Priority: P1)

Как пользователь экрана ивента, я вижу длинную ссылку в поле "Локация" аккуратно и читабельно, без ломания верстки карточки.

**Why this priority**: сейчас это прямой UX-дефект на основном экране деталей ивента.

**Independent Test**: открыть ивент с `locationOrLink` длиной 120+ символов (включая длинные query-параметры) и убедиться, что строка остается внутри карточки и не перекрывает соседние элементы.

**Acceptance Scenarios**:

1. **Given** в `locationOrLink` передана длинная URL-строка, **When** открывается экран `EventDetails`, **Then** значение локации визуально остается в границах строки/карточки и не вызывает горизонтального скролла.
2. **Given** URL содержит длинный непрерывный токен (без пробелов), **When** строка рендерится, **Then** перенос выполняется корректно без наложения на правую границу.
3. **Given** URL слишком длинный для полного показа, **When** строка отображается, **Then** применяется ограничение в 2 строки с троеточием.

---

### User Story 2 - Тап по локации и Google Maps поведение (Priority: P1)

Как пользователь, я могу нажать на ссылку в поле "Локация" и открыть ее. Если это Google Maps ссылка, должен сработать приоритет открытия через Google Maps (если доступно) либо fallback на обычную ссылку.

**Why this priority**: пользователь ожидает, что локация в формате карты будет интерактивной и приведет к построению маршрута.

**Independent Test**: открыть ивент с `locationOrLink=https://maps.app.goo.gl/...`, тапнуть по строке локации и проверить, что открывается карта (через app/universal link) или URL во внешнем браузере.

**Acceptance Scenarios**:

1. **Given** `locationOrLink` является валидной Google Maps ссылкой, **When** пользователь тапает по строке локации, **Then** приложение пытается открыть карту через Google Maps-compatible путь и при недоступности корректно открывает HTTPS-ссылку.
2. **Given** `locationOrLink` является валидной не-google URL, **When** пользователь тапает по строке локации, **Then** открывается соответствующая внешняя ссылка без ошибок UI.
3. **Given** локация не является URL (обычный текст адреса), **When** пользователь открывает экран, **Then** строка рендерится как текст без интерактивного перехода.

---

### User Story 3 - Устойчивость и нормализация ссылок (Priority: P2)

Как пользователь, я не сталкиваюсь с поломкой сценария, даже если ссылка в поле локации введена в нетипичном формате.

**Why this priority**: защищает от реальных пользовательских данных (без протокола, с пробелами, с локальными доменами).

**Independent Test**: протестировать значения `maps.app.goo.gl/...`, `www.google.com/maps/...`, `google.com/maps?...`, `example.com/path`, `Пречистенка 12` и убедиться в предсказуемом поведении.

**Acceptance Scenarios**:

1. **Given** ссылка введена без `http/https` (`www.google.com/maps/...`), **When** пользователь тапает по локации, **Then** система нормализует URL и открывает корректный HTTPS target.
2. **Given** строка содержит невалидный URL, **When** выполняется парсинг, **Then** интерфейс не падает и показывает текст как неинтерактивный.
3. **Given** в URL присутствуют unicode-символы или encoded query, **When** выполняется рендер и открытие, **Then** ссылка отображается и открывается без потери данных.

---

### Edge Cases

- Очень длинный URL (200+ символов) с query-параметрами и без естественных разделителей.
- `locationOrLink` содержит URL в конце текста (например, `Встреча у входа: https://maps...`).
- URL без протокола (`maps.google.com/...`, `www.example.com/...`).
- URL с лишними пробелами в начале/конце.
- Значение с небезопасным протоколом (`javascript:`) — должно быть отклонено как ссылка.
- Открытие ссылки в Telegram WebView, где часть API может быть недоступна.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: Система MUST определять, является ли `locationOrLink` валидной web-ссылкой (`http/https`) после нормализации входного значения.
- **FR-002**: Система MUST поддерживать нормализацию URL без протокола (`www.*`, `google.*`, `maps.*`) до `https://...`.
- **FR-003**: Для строки "Локация" на `EventDetails` система MUST отображать значение в пределах карточки без горизонтального overflow.
- **FR-004**: Для длинного ссылочного значения система MUST использовать перенос внутри слова (`overflow-wrap:anywhere` эквивалент) и ограничение отображения до 2 строк с многоточием.
- **FR-005**: Если `locationOrLink` распознан как URL, строка MUST быть интерактивной по тапу.
- **FR-006**: Если `locationOrLink` не распознан как URL, строка MUST оставаться неинтерактивным текстом.
- **FR-007**: Система MUST распознавать Google Maps ссылки по хостам/паттернам: `maps.app.goo.gl`, `maps.google.*`, `google.*` + путь `/maps`.
- **FR-008**: Для Google Maps ссылок система MUST использовать Google Maps-priority open path (через Telegram SDK openLink с браузерным приоритетом, если доступно), затем fallback на обычное открытие HTTPS URL.
- **FR-009**: Для не-google URL система MUST открывать ссылку через стандартный внешний open flow (Telegram SDK openLink, fallback browser open).
- **FR-010**: Любая ошибка при попытке открытия ссылки MUST обрабатываться без падения экрана и без блокировки дальнейших действий пользователя.
- **FR-011**: Система MUST игнорировать небезопасные схемы (`javascript:`, `data:`) и рендерить их как обычный текст.
- **FR-012**: Фича MUST быть реализована без изменений бэкенд API/контрактов (поле `locationOrLink` остается строкой).
- **FR-013**: Система MUST поддерживать текущее поведение остальных строк `DetailRow` (даты, участники, мин. уровень) без визуальных регрессий.

### Key Entities _(include if feature involves data)_

- **ParsedLocationLink**: нормализованное представление значения `locationOrLink` с полями `raw`, `displayText`, `url`, `kind`, `isClickable`.
- **LocationLinkKind**: тип распознанного значения (`plain_text`, `url`, `google_maps_url`).
- **LocationOpenAction**: результат стратегии открытия ссылки (`open_via_google_priority`, `open_via_default`, `not_openable`).

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: В 100% smoke-кейсов с длинными URL (включая 200+ символов) строка локации не выходит за правую границу карточки и не создает горизонтальный скролл.
- **SC-002**: В 100% smoke-кейсов Google Maps URL распознаются как `google_maps_url` и запускают Google-priority open flow.
- **SC-003**: В 100% smoke-кейсов не-google URL открываются по тапу, а plain text адрес остается неинтерактивным.
- **SC-004**: В 100% smoke-кейсов с невалидными/небезопасными ссылками экран не падает и остается интерактивным.
