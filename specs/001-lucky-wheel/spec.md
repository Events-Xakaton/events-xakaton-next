# Feature Specification: Lucky Wheel for Events

**Feature Branch**: `001-lucky-wheel`  
**Created**: 2026-03-14  
**Status**: Draft  
**Input**: User description: "На экране Главная -> События после серии быстрых свайпов или тапов по разным карточкам появляется секретная кнопка Мне повезет. Она запускает экран колеса фортуны, где есть кнопка \"запустить\". По нажатию на эту кнопку крутится колесо и по итогу останавливается на каком-то ивенте. Ивент должен быть заранее определен из доступных пользователю, однако для пользователя это выглядит так, будто бы выбор происходит случайным образом. Попадать должны только будущие ивенты, где еще есть места (если установлен лимит), и где пользователь еще не является участником. Если ивентов мало, должен выбираться не просто самый первый по времени, а случайный из первых (следующих)."

## Clarifications

### Session 2026-03-14

- Q: Какой порог секрета для появления кнопки «Мне повезет»? → A: Кнопка появляется, если пользователь быстрым скроллом вверх/вниз просмотрел 9 ивентов менее чем за 3 секунды.
- Q: Какой размер окна ближайших событий `K` использовать для выбора? → A: `K=5`.
- Q: Какой лимит частоты срабатывания механики? → A: Не более одного срабатывания в текущий календарный день (сброс по границе дня, не скользящие 24 часа).
- Q: По какой таймзоне определять границу текущего дня? → A: По UTC (граница дня в 00:00 UTC).
- Q: Где фиксировать ограничение «не более 1 раза в день»? → A: Только на сервере, без клиентского кеша лимита.

## User Scenarios & Testing _(mandatory)_

### User Story 1 - Секретный вход в механику (Priority: P1)

Как пользователь ленты ивентов, я могу случайно открыть скрытую кнопку «Мне повезет» после серии быстрых прокруток ленты вверх/вниз, чтобы перейти в механику выбора ивента.

**Why this priority**: без скрытого триггера сама механика не активируется и фича не существует для пользователя.

**Independent Test**: на экране «Главная -> Ивенты» выполнить быструю прокрутку ленты вверх/вниз и проверить появление кнопки «Мне повезет» после просмотра 9 ивентов менее чем за 3 секунды, затем открыть экран колеса.

**Acceptance Scenarios**:

1. **Given** пользователь находится на вкладке «Ивенты» и в ленте есть достаточное число карточек, **When** он быстрым скроллом вверх/вниз просматривает 9 ивентов менее чем за 3 секунды, **Then** появляется скрытая кнопка «Мне повезет».
2. **Given** кнопка «Мне повезет» появилась, **When** пользователь нажимает её, **Then** открывается экран колеса фортуны с кнопкой «Запустить».
3. **Given** пользователь ушел с экрана «Ивенты» или серия действий прервана, **When** условия секрета больше не выполнены, **Then** скрытая кнопка не отображается.

---

### User Story 2 - Прозрачный для системы, случайный для пользователя выбор (Priority: P1)

Как пользователь экрана колеса, я запускаю вращение и получаю один рекомендованный ивент, который заранее выбран системой из допустимых и визуально подан как случайный результат.

**Why this priority**: основной продуктовый результат фичи - помочь пользователю быстро выбрать подходящий ивент.

**Independent Test**: открыть экран колеса, нажать «Запустить», дождаться окончания анимации и убедиться, что открывается/показывается именно допустимый ивент по бизнес-правилам.

**Acceptance Scenarios**:

1. **Given** у пользователя есть допустимые ивенты, **When** он нажимает «Запустить», **Then** система заранее фиксирует конкретный ивент для сессии запуска и показывает анимацию вращения.
2. **Given** вращение завершено, **When** колесо останавливается, **Then** финальный сектор соответствует заранее зафиксированному ивенту.
3. **Given** результат определен, **When** пользователь подтверждает результат, **Then** открываются детали выбранного ивента.

---

### User Story 3 - Корректная выборка кандидатов и отказоустойчивость (Priority: P2)

Как пользователь, я никогда не получаю недоступный ивент (прошедший, переполненный или уже посещаемый мной), а при отсутствии кандидатов получаю понятный сценарий без ошибки.

**Why this priority**: защищает от ложных обещаний, ошибок записи и деградации доверия к механике.

**Independent Test**: подготовить данные с комбинацией недопустимых и допустимых ивентов, проверить фильтрацию, поведение при 0 кандидатов и логику выбора из ближайших ивентов.

**Acceptance Scenarios**:

1. **Given** в системе есть ивенты с разными статусами и вместимостью, **When** формируется пул кандидатов, **Then** попадают только будущие (`upcoming`) ивенты, где пользователь не участник и где есть свободные места при ограничении `maxParticipants`.
2. **Given** количество кандидатов больше окна «ближайших», **When** выбирается результат, **Then** выбор происходит случайно из ограниченного окна ближайших по времени, а не из всего списка.
3. **Given** количество кандидатов меньше или равно окну «ближайших», **When** выбирается результат, **Then** выбор происходит случайно из всех доступных кандидатов, а не всегда первый по времени.
4. **Given** кандидатов нет, **When** пользователь запускает колесо, **Then** фронт показывает понятное сообщение и CTA на возврат в ленту без падения сценария.
5. **Given** пользователь уже использовал механику в текущий календарный день, **When** он повторно нажимает «Запустить», **Then** backend возвращает `DAILY_LIMIT_REACHED`, а фронт показывает понятный fallback без повторного запуска.

---

### Edge Cases

- Пользователь медленно скроллит ленту: секрет не должен сработать, если 9 ивентов не просмотрены за период менее 3 секунд.
- Пользователь нажимает «Запустить» несколько раз подряд: должен обрабатываться только один активный запуск.
- Кандидат стал недоступен между префиксом выбора и открытием деталей (например, место заняли): пользователь получает корректное сообщение и повторный запуск.
- В ленте мало ивентов (1-2): колесо и выбор работают без деградации UX.
- Часовые пояса не должны ошибочно переводить `upcoming` в `past/ongoing`.
- Пользователь выполнил запуск до 00:00 UTC и повторил после 00:00 UTC: ограничение должно сбрасываться по новой UTC-дате.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: Система MUST отслеживать на экране «Главная -> Ивенты» последовательность быстрых прокруток ленты (вверх/вниз) и факт просмотра карточек ивентов в viewport.
- **FR-002**: Система MUST показывать скрытую кнопку «Мне повезет» только если пользователь просмотрел 9 ивентов менее чем за 3 секунды за счет быстрого скролла ленты вверх/вниз.
- **FR-003**: Система MUST открывать отдельный экран колеса фортуны по нажатию кнопки «Мне повезет».
- **FR-004**: Экран колеса MUST содержать кнопку «Запустить» и анимацию вращения с финальной остановкой на заранее определенном результате.
- **FR-005**: При запуске вращения система MUST запрашивать/получать заранее определенный результат до начала визуальной фазы остановки колеса.
- **FR-006**: Бэкенд MUST формировать пул кандидатов только из событий со статусом `upcoming` на момент выбора.
- **FR-007**: Бэкенд MUST исключать из пула кандидатов события, где пользователь уже участвует (`joined`).
- **FR-008**: Бэкенд MUST исключать события без свободных мест, если для события задан `maxParticipants` и достигнут лимит.
- **FR-009**: Бэкенд MUST сортировать допустимых кандидатов по времени начала (`startsAtUtc` asc), брать окно ближайших `K=5` кандидатов и выбирать случайно внутри этого окна.
- **FR-010**: Если допустимых кандидатов меньше `K`, система MUST выбирать случайно из всех допустимых кандидатов.
- **FR-011**: Если допустимых кандидатов нет, API MUST возвращать доменную ошибку «нет подходящих событий», а фронт MUST показывать дружественное сообщение без крэша.
- **FR-012**: После завершения вращения фронт MUST открыть экран деталей выбранного ивента по его `id`.
- **FR-013**: Система MUST отправлять аналитику по этапам: раскрытие секрета, запуск колеса, показ результата, переход в детали.
- **FR-014**: Параметры секрета (порог просмотренных ивентов и временное окно) MUST быть конфигурируемыми константами приложения; `K` поддерживается как параметр политики выбора с дефолтом `5`.
- **FR-015**: Система MUST ограничивать срабатывание механики `Мне повезет` до одного раза на пользователя в текущий календарный день по UTC, со сбросом в `00:00 UTC` (не через 24 часа от последнего запуска).
- **FR-016**: Ограничение по частоте срабатывания MUST проверяться и храниться только на сервере; клиент MUST NOT быть источником истины для этого ограничения.

### Key Entities _(include if feature involves data)_

- **LuckyTriggerSession**: Эфемерное состояние на фронте для отслеживания количества просмотренных карточек при быстром скролле, временного окна и флага раскрытия кнопки.
- **LuckyWheelRun**: Сессия одного запуска колеса с полями `startedAt`, `resolvedEventId`, `animationState`, `isLocked`.
- **LuckyCandidateEvent**: Проекция события для выбора колеса: `eventId`, `startsAtUtc`, `freeSpots`, `joinedByMe`, `computedStatus`.
- **LuckySelectionPolicy**: Набор конфигов и правил (`K`, фильтры допустимости, стратегия random-within-nearest-window).

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: В 100% запусков колеса выбранный ивент удовлетворяет фильтрам допустимости: `upcoming`, не `joinedByMe`, и с местами при лимите.
- **SC-002**: В 95% сессий на устройстве со средним FPS не ниже 50 анимация завершения колеса укладывается в 2-4 секунды без визуальных рывков.
- **SC-003**: При повторных прогонах на одном и том же наборе из `<=K` ближайших кандидатов распределение результатов не вырождается в постоянный выбор первого ивента.
- **SC-004**: При отсутствии кандидатов 100% пользователей получают понятный fallback-сценарий (сообщение + возврат в ленту) без технической ошибки на экране.
