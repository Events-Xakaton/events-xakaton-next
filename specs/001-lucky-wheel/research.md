# Research: Lucky Wheel for Events

## Decision 1: Секретный триггер считать по просмотрам карточек в viewport

- Decision: Триггер раскрытия кнопки считается по уникальным появлениям карточек в viewport во время быстрого скролла, порог `9 карточек < 3с`.
- Rationale: Это напрямую соответствует уточнению пользователя про "пролистано 9 ивентов" и не зависит от тапов.
- Alternatives considered:
  - Считать только scroll events: шумно и трудно связать с фактом просмотра карточек.
  - Считать только snap-позиции: может пропускать реальные быстрые просмотры.

## Decision 2: Ограничение 1 запуск/день держать только на сервере

- Decision: Сервер хранит и проверяет дневной лимит; клиент не кэширует лимит как источник истины.
- Rationale: Исключает обход через очистку стейта/переустановку клиента и соответствует clarify-решению.
- Alternatives considered:
  - Только клиент: легко обойти.
  - Сервер + клиентский кеш: быстрее UX, но добавляет риск рассинхрона и лишнюю сложность.

## Decision 3: Граница дня по UTC

- Decision: Для проверки лимита сервер использует UTC-день (`00:00 UTC`), без учета пользовательской таймзоны.
- Rationale: Это максимально простая и однозначная реализация без зависимости от качества timezone-данных пользователя.
- Alternatives considered:
  - Локальная таймзона пользователя: более “человечно”, но сложнее и рискованнее из-за форматов TZ.
  - Таймзона сервера: зависит от инфраструктуры, а не от продукта.

## Decision 4: Выбор ивента через окно ближайших `K=5`

- Decision: Из eligible upcoming-событий берутся ближайшие по `startsAtUtc`, окно `K=5`, затем случайный выбор внутри окна.
- Rationale: Баланс между релевантностью (ближайшие) и разнообразием (не всегда первый).
- Alternatives considered:
  - Полностью случайно по всему списку: может дать слишком дальний ивент.
  - Всегда первый: нет ощущения случайности.

## Decision 5: Контракт random endpoint расширить кодами причин отказа

- Decision: Для `/events/random` оставить успешный ответ `{ id }`, а в отказах использовать `404` с машинно-читаемым кодом (`NO_ELIGIBLE_EVENTS`, `DAILY_LIMIT_REACHED`) без дополнительных query-параметров.
- Rationale: Фронт может показать корректный fallback для разных отказов без хрупкого парсинга текста.
- Alternatives considered:
  - Только человекочитаемые тексты: сложнее стабильно обрабатывать на клиенте.
  - 400 вместо 404: хуже семантика для "ничего не найдено/недоступно".
