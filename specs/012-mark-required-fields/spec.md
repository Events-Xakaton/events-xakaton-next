# Feature Specification: Required Field Markers on Create Screens

**Feature Branch**: `012-mark-required-fields`  
**Created**: 2026-03-15  
**Status**: Implemented (build-validated on frontend)  
**Input**: User description: "На экранах создания ивентов и клубов есть обязательные поля. Необходимо пометить их обязательность, иначе пользователь не понимает почему неактивна кнопка создания. Сделай через spec-kit."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Event required fields are clearly marked (Priority: P1)

Как пользователь экрана создания мероприятия, я сразу вижу, какие поля обязательны, чтобы понимать, почему кнопка создания может быть неактивной.

**Why this priority**: Без этого пользователь не понимает причину disabled-состояния кнопки и не может завершить создание мероприятия без проб и ошибок.

**Independent Test**: открыть вкладку создания мероприятия и визуально проверить наличие маркеров обязательности у всех обязательных полей.

**Acceptance Scenarios**:

1. **Given** открыта вкладка создания мероприятия, **When** пользователь смотрит форму, **Then** обязательные поля помечены визуальным маркером `*`.
2. **Given** пользователь не заполнил одно из обязательных полей, **When** кнопка создания неактивна, **Then** по форме понятно, какие поля обязательны к заполнению.
3. **Given** пользователь заполнил обязательные поля корректно, **When** форма валидна, **Then** кнопка создания активируется как и раньше.

---

### User Story 2 - Club required fields are clearly marked (Priority: P1)

Как пользователь экрана создания клуба, я вижу обязательные поля и понимаю, что нужно заполнить для активации кнопки создания.

**Why this priority**: Проблема UX идентична форме ивента и напрямую влияет на завершение создания клуба.

**Independent Test**: открыть вкладку создания клуба и проверить наличие маркеров обязательности у названия и описания.

**Acceptance Scenarios**:

1. **Given** открыта вкладка создания клуба, **When** пользователь смотрит форму, **Then** обязательные поля помечены визуальным маркером `*`.
2. **Given** не заполнено название или описание, **When** кнопка создания неактивна, **Then** пользователь видит, что эти поля обязательны.
3. **Given** название и описание заполнены, **When** форма валидна, **Then** кнопка создания активируется как и раньше.

---

### Edge Cases

- Переключение между вкладками `Мероприятие` и `Клуб` не должно ломать/скрывать маркеры обязательных полей.
- Поля, не влияющие на валидность формы (например, лимит участников, возраст, минимальный уровень), не должны быть помечены как обязательные.
- Обязательные поля с пробелами должны считаться незаполненными по действующей логике валидации.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST визуально помечать обязательные поля формы создания мероприятия.
- **FR-002**: Система MUST визуально помечать обязательные поля формы создания клуба.
- **FR-003**: Система MUST отображать понятную подсказку, что `*` означает обязательное поле.
- **FR-004**: Система MUST сохранять существующую логику валидации и активации кнопок создания (без изменения бизнес-правил).
- **FR-005**: Система MUST не помечать необязательные поля как обязательные.

### Key Entities *(include if feature involves data)*

- **RequiredFieldLabel**: UI-обозначение обязательного поля в виде текстового маркера `*` рядом с именем поля.
- **RequiredFieldsHint**: текстовая подсказка в форме о том, что поля с `*` обязательны.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: На обеих формах (`event`, `club`) 100% обязательных полей визуально отмечены маркером `*`.
- **SC-002**: На обеих формах присутствует заметная подсказка о значении маркера `*`.
- **SC-003**: Поведение кнопок `Создать мероприятие` и `Создать клуб` в части disabled/enabled не меняется относительно текущей валидации.
